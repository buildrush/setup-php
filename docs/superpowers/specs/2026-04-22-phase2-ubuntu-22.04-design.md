# Phase 2 — Slice D: ubuntu-22.04 Runner Coverage (Phase-2 Exit)

**Status:** Design, awaiting implementation plan
**Date:** 2026-04-22
**Supersedes:** Nothing (final Phase-2 slice per `2026-04-17-phase2-compat-slice-design.md` §8)
**Target release:** Next patch alpha (e.g. `v0.2.0-alpha.12`). Promoting to the `v0.2.0-alpha` exit tag is a release-please config concern tracked separately.

## 1. Summary

Final Phase-2 slice. Moves every bundle build from `ubuntu-24.04` (noble) to `ubuntu-22.04` (jammy) so compiled bundles run on both OS releases via glibc forward-compat. After merge, the bundle matrix stays at 182 entries but every digest is jammy-compiled; users can run the action on `ubuntu-22.04` or `ubuntu-24.04` runners interchangeably.

One explicit gap: `imagick` links against ImageMagick 6 with different SOVERSIONs on jammy (`.so.6`) and noble (`.so.7`). A jammy-compiled imagick `.so` cannot `dlopen` the noble lib. imagick stays in the catalog — works on jammy (primary target), broken on noble — with a tracking issue filed on merge for cross-OS imagick support (via bundled libs or per-OS schema, whichever a follow-up picks).

The work is bounded but touches every layer for the first time since C1:
- `internal/planner` gains a `BUILDER_OS` hash input so switching runners invalidates all 182 spec_hashes.
- Workflows (`build-php-core.yml`, `build-extension.yml`, `compat-harness.yml`) swap `ubuntu-24.04` → `ubuntu-22.04` in `runs-on` expressions; harness also gains `runner_os` fixture dimension.
- All 18 PECL YAMLs' `runtime_deps.linux` drop noble-specific `-t64` suffixes for jammy-native names; noble's apt resolves those via transitional packages (verified in-CI).
- 5 new `bare-jammy-<NN>` fixtures, x86_64-only, prove jammy-built bundles load on 22.04 runners.

Wall-clock dominated by 10 grpc builds (5 versions × 2 archs) ≈ 45–60 min matrix-parallel — same ceiling seen in slice C2b.

## 2. Goals & non-goals

### Goals
- `builders/common/builder-os.env` exists containing a single variable `BUILDER_OS=ubuntu-22.04`.
- `internal/planner.ComputeSpecHash` folds `BUILDER_OS` into the hash so changing that file invalidates every cell's spec_hash.
- `build-php-core.yml` and `build-extension.yml` dispatch to `ubuntu-22.04` (amd64) / `ubuntu-22.04-arm` (arm64).
- `runtime_deps.linux` across all 18 PECL extensions lists jammy-native package names — verified present on noble via transitional packages for every dep except imagick's ImageMagick pair (which has no noble transitional).
- 5 new `bare-jammy-<NN>` fixtures exist in `test/compat/fixtures.yaml` (one per PHP minor, x86_64 only), each with `runner_os: 'ubuntu-22.04'`.
- `compat-harness.yml`'s `ours` / `theirs` jobs dispatch to `ubuntu-22.04` / `ubuntu-22.04-arm` when `matrix.runner_os == 'ubuntu-22.04'`, else to the noble runners.
- `bundles.lock` holds 182 entries (same count) after pipeline completes — every digest refreshes to a jammy-compiled value.
- `cmd/phpup/main.go` runtime catalog map's `RuntimeDeps` values mirror the catalog's jammy-rewritten values.
- `README.md` status line documents "Linux x86_64 + aarch64 on Ubuntu 22.04 and 24.04" and flags the imagick-on-noble gap with an issue link.

### Non-goals
- Per-OS bundle keying (expanding `bundles.lock` key schema). Approach rejected earlier: too expensive for single-gap case.
- Bundling runtime libs inside extension bundles (product-vision §6.1 aspiration). Deferred to a later slice if imagick cross-OS becomes a priority.
- Arm64 jammy harness fixtures. Omitted to hold harness CI cost; if a user issue surfaces arm64-jammy drift, a follow-up slice adds `bare-jammy-arm64-<NN>`.
- macOS / Windows support — Phases 5 / 6.
- PECL version bumps — single-purpose slices per project convention.
- New compat harness axes (per-coverage, per-ini-file) on jammy — `bare-jammy-<NN>` is the minimum-viable smoke set.
- Cutting the `v0.2.0-alpha` exit tag (without dot-suffix). That's a release-please config change orthogonal to the engineering work; tracked separately.

## 4. Architecture

### 4.1 Component touchpoints

| Layer | Change |
|---|---|
| `builders/common/builder-os.env` | **New.** One line: `BUILDER_OS=ubuntu-22.04`. Sourced by both Go and shell. |
| `internal/planner/planner.go` | `ComputeSpecHash` reads `BUILDER_OS` from that file, folds into hash alongside `catalogData` and `builderHash`. |
| `internal/planner/planner_test.go` | Golden test: two hashes computed with different `BUILDER_OS` values differ; absent file produces a clear error (mirrors `HashFile` contract). |
| `.github/workflows/build-php-core.yml` | `runs-on: ${{ inputs.arch == 'aarch64' && 'ubuntu-22.04-arm' || 'ubuntu-22.04' }}`. |
| `.github/workflows/build-extension.yml` | Same runner map. |
| `.github/workflows/compat-harness.yml` | `ours` + `theirs` `runs-on` becomes `${{ matrix.arch == 'aarch64' && (matrix.runner_os == 'ubuntu-22.04' && 'ubuntu-22.04-arm' || 'ubuntu-24.04-arm') || (matrix.runner_os == 'ubuntu-22.04' && 'ubuntu-22.04' || 'ubuntu-24.04') }}`. |
| `catalog/extensions/*.yaml` (18 files) | `runtime_deps.linux` values rewritten per §3.1 table. |
| `cmd/phpup/main.go` runtime catalog map | `RuntimeDeps` values mirror catalog changes. Hand-synced (tracked tech debt since B1). |
| `test/compat/fixtures.yaml` | 5 new `bare-jammy-<NN>` fixtures, each with `runner_os: 'ubuntu-22.04'`, `arch` absent (defaults `x86_64`). |
| `README.md` | Status line updates; imagick-on-noble gap footnote. |
| `internal/catalog`, `internal/resolve`, `internal/compose`, rest of `internal/` | **Unchanged.** |

### 4.2 `ComputeSpecHash` BUILDER_OS input (chosen idiom)

Three alternatives considered:

- **Env file, sourced by both Go and shell** (chosen). `builders/common/builder-os.env` with `BUILDER_OS=ubuntu-22.04`. Go reads the file and writes the variable value to the hash. Shell in workflows could source it if needed (not required; `runs-on` uses the value literally in the workflow). Single source of truth; one-line change to bump the pinned OS in the future.
- **Hardcoded constant in Go.** Simpler but hides the pinning from workflow visibility; future OS bump touches Go code + workflow files separately.
- **Workflow input variable with a default.** Couples `ComputeSpecHash` to workflow-input semantics; invariant goes through CI config rather than a repo-local file. Rejected for testability.

The chosen idiom extends the existing "reads a pinned file" pattern already used for `builderHash` (which hashes `builders/linux/build-*.sh`).

### 4.3 Harness `runner_os` fixture field (chosen schema)

New optional field `runner_os: string` on each fixture entry. Absent → defaults to `ubuntu-24.04`. `compat-harness.yml`'s fixture-matrix generator (`yq | jq`) passes every field through to `matrix.*` unchanged, so the new field flows to `matrix.runner_os` with zero generator edit. The `runs-on` expression pivots on `matrix.runner_os × matrix.arch` per §4.1.

Lockfile-presence filter already introduced in slice A (`(version, arch)` tuples) does **not** need to gain a `(version, arch, runner_os)` dimension — the bundle isn't keyed by runner OS; the bundle keying stays `php:<ver>:linux:<arch>:<ts>`. A fixture with `runner_os: ubuntu-22.04` just runs on a different host; it resolves the same bundle as a `runner_os: ubuntu-24.04` fixture for the same (version, arch).

### 4.4 imagick gap disposition

The jammy-compiled imagick `.so` links against `libMagickWand-6.Q16.so.6`. On noble this file isn't present (noble's `libmagickwand-6.q16-7t64` provides `.so.7`), and no transitional back-port exists. Three response options considered, all deferred to a follow-up:

- Bundle `libMagickWand-6.Q16.so.6` + dependencies inside the imagick bundle (product-vision §6.1 aspiration).
- Add per-OS bundle keying for imagick only (hybrid schema change).
- Remove imagick from the catalog until one of the above lands.

For this slice: imagick stays in the catalog at its current jammy-compatible form. Users on jammy runners get working imagick. Users on noble runners hit a clear `dlopen` error at `installRuntimeDeps` + compose time (noble's apt can't install `libmagickwand-6.q16-6`, so the failure surfaces early rather than at PHP startup). Issue filed on merge with title "imagick cross-OS support (ImageMagick SOVERSION drift jammy↔noble)" and a link back to this spec.

### 4.5 Force-rebuild behavior

`ComputeSpecHash` grows to `sha256(catalogData || builderHash || BUILDER_OS || cellIdentity)`. Flipping `BUILDER_OS` from its default (the implicit `""` empty for existing bundles — since the field is new) to `ubuntu-22.04` changes the hash of every cell. All 182 existing lockfile entries' `spec_hash` fields mismatch → planner emits every cell to the build matrix. 182 × ~5–10 min matrix-parallel on the `max-parallel: 30` limit → ~45–60 min wall-clock dominated by grpc cells.

The rebuild produces 182 new digests (content-different — jammy vs noble glibc / toolchain versions). Bundle count in `bundles.lock` is unchanged (lockfile keys are identical); only the `digest` and `spec_hash` fields refresh.

### 4.6 Lockfile mutation scale

Bot lockfile commit will touch **182 lockfile entries**. Each entry's `digest` and `spec_hash` fields change. This is the largest single lockfile diff of Phase 2. `bundles.lock` itself stays under 100 KB; the diff is large but structured.

## 5. Testing

### 5.1 Unit tests

- `internal/planner/planner_test.go` — `TestComputeSpecHash_BuilderOSIsLoadBearing`: build two `MatrixCell`s with identical `catalogData` + `builderHash` but different `BUILDER_OS` strings; assert hashes differ. Also `TestComputeSpecHash_BuilderOSMissingFile`: when `builders/common/builder-os.env` doesn't exist, `ComputeSpecHash`'s caller bubbles up a clear error (mirrors the existing `HashFile` missing-file contract).
- No other packages need test changes.

### 5.2 Build-time smoke (per PHP core)

Every one of the 10 core bundles runs its `smoke:` block after jammy compile. Catches any jammy-vs-noble compile-time surprise per cell per arch. If PHP 8.1 on arm64-jammy fails to configure (unlikely — jammy has gcc 11, compatible with every supported PHP), apply `exclude: { arch: aarch64, php: "8.1" }` on the `"8.1"` catalog block + file tracking issue.

### 5.3 Harness fixtures

5 new `bare-jammy-<NN>` fixtures (one per PHP minor, x86_64 only). Each runs `ours` and `theirs` on `ubuntu-22.04`. Primary signals:

- jammy-built PHP core loads on 22.04 runner.
- `installRuntimeDeps` picks up the jammy-named apt packages.
- `php -v` / `php -m` / ini probes match v2's baseline on 22.04.

The existing 65 fixtures default to `runner_os: ubuntu-24.04` and load the same jammy-rebuilt bundles — they prove the "jammy-built bundle also runs on noble" side of the contract. Expected failures: any extension whose jammy-native lib isn't resolvable on noble. Based on §3.1 table only imagick should hit this, and only on noble.

### 5.4 Cross-slice regression

The 5 `bare-arm64-<NN>` fixtures from C1 continue to run on `ubuntu-24.04-arm` (default `runner_os`). The 5 `multi-ext-top14-arm64-<NN>` from C2a + 5 `multi-ext-hard4-arm64-<NN>` from C2b stay on `ubuntu-24.04-arm`. All load jammy-rebuilt bundles; failures (if any) surface here.

### 5.5 Coverage

Per `CLAUDE.md`: 80% per-package. `internal/planner` gains a small `BUILDER_OS` branch + test. Everything else unchanged. Numbers held by construction.

### 5.6 Local development

`make bundle-php PHP_VERSION=8.4 ARCH=x86_64` still works via Docker with an `ubuntu:22.04` image. Update the Makefile's `bundle-php` / `bundle-ext` targets to use `ubuntu:22.04` instead of `ubuntu:24.04` — single image-tag change. Document the runner migration in the contribution section.

## 7. Open questions

Resolved in-PR per standard playbook:

1. **`ubuntu-22.04-arm` runner availability.** Standard label per GitHub docs. If it doesn't resolve on the first CI run, fall back strategies (per §7 option list in brainstorm): build arm64 on 24.04-arm with explicit `GLIBC_22` target flags, or use qemu / Docker arm64 emulation on amd64. Decide based on first-run error text.
2. **Noble transitional resolution per package.** §3.1 claims every jammy name except imagick's pair resolves on noble via transitional. If any one doesn't, fix by listing both variants in that extension's `runtime_deps.linux` (apt picks whichever is present) — non-blocking fix, done in-PR.
3. **imagick response.** Documented as known gap. No runtime-code branching in this slice.
4. **grpc 10-cell jammy rebuild.** One-time cost. Monitoring the first run verifies wall-clock is within budget (~60 min ceiling).
5. **`BUILDER_OS` env file reading in planner tests.** Tests should use `t.TempDir()` + a synthesized env file rather than reading the real repo file. Pattern matches existing `HashFile` test usage.
6. **Node-20 deprecation warning.** Unchanged from prior slices. Not a D concern.
